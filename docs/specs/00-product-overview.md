# Product Overview

## Purpose

Test Design Studio は、テスト対象アプリケーションの仕様理解とテスト設計を一体で行うためのローカルファーストなワークスペースである。

単なるテストケース管理ではなく、次の情報を構造化してつなげることを目的とする。

- 機能・ユースケース
- 画面
- UI要素
- データモデル
- データ種別
- 状態
- 状態遷移
- 業務ルール
- エラー・例外
- 未確認事項
- テスト観点
- テストケース
- 変更履歴
- トレーサビリティ

## Target users

主な利用者は、Webアプリケーションの仕様把握、回帰テスト設計、テスト自動化準備を行うQAエンジニア、テスト設計者、開発者である。

特に次の課題を持つ利用者を対象とする。

- 既存のテスト項目書が肥大化し、何をどこまで確認しているか分かりにくい。
- 仕様、観点、ケース、変更履歴が分散しており、正本が見えない。
- UI変更時に、影響する観点やテストケースを追いにくい。
- 手動テストと自動化候補の切り分けが属人的になっている。
- AIエージェントに渡せる構造化済みの仕様情報が不足している。

## Product position

Test Design Studio は、次の中間に位置する。

- 要件定義ツール
- テスト設計ツール
- テストケース管理ツール
- テスト自動化準備ツール
- 仕様ナレッジベース

ただし、初期フェーズでは包括的なALMやクラウド型テスト管理ツールを目指さない。ローカルファーストで、1機能分の仕様把握から観点・ケース作成までを確実に行えることを優先する。

## Core value

このプロダクトの中核価値は、テストケースそのものではなく、テストケースに至る前提を残せることである。

具体的には、次の問いに答えられる状態を目指す。

- このテスト観点は、どの仕様・UI・業務ルールから来ているか。
- このテストケースは、どの観点を確認しているか。
- このUIが変わった場合、どの観点・ケースに影響するか。
- どの仕様が未確認のままテスト設計に使われているか。
- 自動化しやすいケースと、人手で確認すべきケースはどれか。

## Product principles

- ローカルファーストであること。
- テスト設計上意味のある構造を扱うこと。
- DOMをそのまま保存するのではなく、仕様理解に必要なUI要素へ整理すること。
- AI生成よりも、根拠のある構造化データを優先すること。
- 変更履歴と影響範囲を追えること。
- MVPを1回で削り切るのではなく、P0、P1、P2で段階的に全体構想へ近づけること。
- 実装フェーズを分け、1PRで完了可能な単位に落とすこと。

## Staged MVP definition

このプロダクトでは、MVPを「最終構想の一部だけを捨てた簡易版」とは扱わない。

MVPは、実務で使える最小価値を段階的に積み上げ、最終的に親設計書の主要構想へ到達するための実装単位である。

### P0: Web Design MVP

P0では、Webアプリ単体で1機能分の仕様把握とテスト設計を完了できる状態を作る。

P0の成功条件:

1. Projectを作成できる。
2. 1機能について、Feature、Screen、主要UiNode、DataType、BusinessRule、OpenQuestionを登録できる。
3. 登録した仕様情報から、TestViewpointとTestCaseを作成できる。
4. TestCaseの手順は、将来のPlaywright draft exportに備えて構造化されたTestStepとして保存できる。
5. 最低限のTraceLinkを保存でき、観点・ケースが元になった仕様要素へ紐づいている。
6. Markdown / JSON として外部出力できる。
7. JSONから新規Projectとして復元できる。

P0では、状態・状態遷移・フロー・エラー・例外はモデルとして予約する。ただし、専用UIや技法生成までは必須にしない。

### P1: Capture & Trace MVP

P1では、実画面から設計素材を取り込み、変更履歴と影響追跡に接続できる状態を作る。

P1の成功条件:

1. Chrome拡張から現在画面のURL、タイトル、選択要素をUI候補として取り込める。
2. DomCaptureCandidateをWebアプリ側で確認・編集し、UiNodeへ変換できる。
3. ChangeRecordを登録できる。
4. TraceLinkを手動で作成・確認できる。
5. 変更対象から影響候補を表示し、人が影響有無を確定できる。

### P2: Assistive Design MVP

P2では、構造化済みデータからテスト設計・AIレビュー・Playwright実装支援へ進む。

P2の成功条件:

1. DataTypeやBusinessRuleから観点候補を生成できる。
2. StateTransitionやDecisionTableを使った観点候補を作れる。
3. 生成候補は自動採用せず、利用者がレビューしてTestViewpoint化できる。
4. AIエージェント向けの構造化context exportを作れる。
5. TestCaseとUiNodeの構造からPlaywright draft exportを作れる。

## Non-goal summary

P0では、次を目的にしない。

- 完全なテスト管理SaaS
- DOMからの仕様完全自動生成
- AIによるテストケース大量生成
- Playwrightコードの完全自動生成
- Playwrightの実行・結果収集・自己修復
- クラウド同期
- 複数ユーザーの同時編集
- Figma / Notion / Jira / GitHub 連携

P1、P2で追加する機能も、正本データを人が確認・編集できることを前提に段階実装する。
