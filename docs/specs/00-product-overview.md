# Product Overview

## Purpose

Test Design Studio は、テスト対象アプリケーションの仕様理解とテスト設計を一体で行うためのローカルファーストなワークスペースである。

単なるテストケース管理ではなく、次の情報を構造化してつなげることを目的とする。

- 機能
- 画面
- UI要素
- データモデル
- データ種別
- 状態
- 業務ルール
- 未確認事項
- テスト観点
- テストケース
- 変更履歴
- トレーサビリティ

## Target users

主な利用者は、Webアプリケーションの仕様把握、回帰テスト設計、テスト自動化準備を行うQAエンジニア、テスト設計者、開発者である。

特に次の課題を持つ利用者を対象とする。

- 既存のテスト項目書が肥大化し、何をどこまで確認しているか分かりにくい。
- 仕様、観点、ケース、変更履歴が分散しており、正本が見えない。
- UI変更時に、影響する観点やテストケースを追いにくい。
- 手動テストと自動化候補の切り分けが属人的になっている。
- AIエージェントに渡せる構造化済みの仕様情報が不足している。

## Product position

Test Design Studio は、次の中間に位置する。

- 要件定義ツール
- テスト設計ツール
- テストケース管理ツール
- テスト自動化準備ツール
- 仕様ナレッジベース

ただし、初期フェーズでは包括的なALMやクラウド型テスト管理ツールを目指さない。ローカルファーストで、1機能分の仕様把握から観点・ケース作成までを確実に行えることを優先する。

## Core value

このプロダクトの中核価値は、テストケースそのものではなく、テストケースに至る前提を残せることである。

具体的には、次の問いに答えられる状態を目指す。

- このテスト観点は、どの仕様・UI・業務ルールから来ているか。
- このテストケースは、どの観点を確認しているか。
- このUIが変わった場合、どの観点・ケースに影響するか。
- どの仕様が未確認のままテスト設計に使われているか。
- 自動化しやすいケースと、人手で確認すべきケースはどれか。

## Product principles

- ローカルファーストであること。
- テスト設計上意味のある構造を扱うこと。
- DOMをそのまま保存するのではなく、仕様理解に必要なUI要素へ整理すること。
- AI生成よりも、根拠のある構造化データを優先すること。
- 変更履歴と影響範囲を追えること。
- 実装フェーズを分け、1PRで完了可能な単位に落とすこと。

## Success criteria

初期バージョンでは、次ができれば成功とする。

1. 1機能について、関連画面、主要UI、データ、業務ルールを登録できる。
2. 登録した仕様情報から、テスト観点とテストケースを作成できる。
3. 観点とケースが、元になった仕様要素へ紐づいている。
4. 変更履歴を登録し、影響する観点・ケースを追える。
5. Markdown / JSON として外部出力できる。
6. Chrome拡張から現在画面のURL、タイトル、選択要素をUI候補として取り込める。

## Non-goal summary

初期フェーズでは、次を目的にしない。

- 完全なテスト管理SaaS
- DOMからの仕様完全自動生成
- AIによるテストケース大量生成
- Playwrightコードの完全自動生成
- クラウド同期
- 複数ユーザーの同時編集
- Figma / Notion / Jira / GitHub 連携
